# 演示与答辩提纲

## 60 秒演示脚本

| 时间 | 操作 | 讲解重点 |
|---|---|---|
| 0–8s | 展示 CPU、每核、内存与历史曲线 | `/proc/stat` 与 `/proc/meminfo` 手写解析 |
| 8–16s | 展示网络、磁盘实时速率 | 累计计数差分、首帧和计数回退处理 |
| 16–25s | 切换 flat/tree，搜索进程并打开详情 | PPID 构树，详情按需读取避免 O(N) 额外 I/O |
| 25–36s | new → 运行 fork 10s | 真实 libc fork、execv、wait4；事件轨显示 PID |
| 36–46s | 启动 `sleep 20`，调整 nice | `exec.Command` 与 `setpriority(2)` |
| 46–54s | 对 sleep 发送 SIGTERM | `kill(2)`、PID+starttime 身份校验 |
| 54–60s | 展示测试命令和评分对照表 | race、vet、解析器边界测试与工程规范 |

## 建议答辩开场

“本项目不是调用系统信息库，而是直接解析 Linux `/proc`。后端把内核累计计数转换成采样区间速率，前端通过 Wails 的 GTK/WebKitGTK 桌面容器以 btop 风格展示；进程操作同时校验 PID 和启动时间，避免 PID 复用误操作。”

## 常见问题

### 为什么不用 gopsutil？

题目的关键技术是理解 `/proc` 字段和 OS 原理。第三方库会隐藏字段映射、采样差分、单位换算和异常处理，不利于展示课程知识点。

### 为什么第一次 CPU/网络/磁盘速率是 0？

文件中是自启动以来的累计量，使用率和速率必须至少有两次样本才能计算差值。首帧没有前一个样本，返回 0 是正确行为。

### 为什么进程 CPU 可能超过 100%？

采用 top 的 Irix 模式：一个逻辑核满载等于 100%，多线程进程可同时占用多个核。

### 为什么 fork 后必须立即 exec？

fork 只复制当前线程，多线程 Go runtime 的其他线程消失，锁和 GC 状态可能不一致。子进程只执行 `execv` 或 `_exit`，用新进程映像替换不安全的 runtime 状态。

### Wails 是否满足 GTK+ 图形界面要求？

Linux 版本的原生窗口和事件循环由 GTK/WebKitGTK 提供，Wails 负责 Go 与界面通信。报告同时说明这不是直接使用 GTK 控件的方案及其权衡。

### 为什么某些进程详情显示 N/A？

Linux 的 ptrace、hidepid 和用户权限会限制 `/proc/[pid]/io`、exe、cwd 或 fd。项目将其视为正常权限边界，不用 root 绕过。

## 演示前检查

- 使用 `wails dev -tags webkit2_41`，确认窗口不是浏览器页面。
- 准备一个允许当前用户控制的 `sleep` 进程，不操作系统服务。
- 录屏分辨率建议 1280×800，字号保持默认。
- 确认事件轨已显示 `/proc online`，网络无流量时可主动执行一次正常下载再观察曲线。
